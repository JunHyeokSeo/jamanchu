# taste-match 기획 분석 리포트

> 분석일: 2026-04-02 | 분석자: planner

---

## 1. 컨셉 적합성 분석

### 핵심 플로우: 익명 취향 공유 → 리액션 → 매칭 → 1:1 대화

**평가: B+ (양호하나 개선 여지 있음)**

taste-match의 핵심 컨셉은 "취향 기반 익명 소셜 매칭"이다. 외모, 스펙이 아닌 **취향의 공감도**를 매칭 기준으로 삼는다는 점에서 컨셉 자체는 명확하고 매력적이다.

**자연스러운 부분:**
- 익명 진입 장벽이 거의 없음 (버튼 한 번으로 자동 닉네임 생성, `UserService.createAnonymousUser()`)
- "취향을 공유한다 → 공감 반응을 받는다"는 SNS적 보상 루프가 존재
- 리액션 중 SAME_HERE("나도!")와 EMPATHY("공감")가 매칭 시그널로 사용되는 설계는 직관적

**부자연스러운 부분:**
- **매칭 트리거 시점이 수동적**: `MatchController.matches()`에서 페이지 진입 시 `calculateMatches()`를 호출함. 사용자가 매칭 탭을 클릭해야만 매칭이 계산되므로, "자연스럽게 매칭된다"는 경험이 아니라 "매칭 탭을 눌러야 결과를 볼 수 있다"는 경험이 됨
- **매칭 → 대화 사이의 동기 부족**: 매칭되었다는 사실만 알려주고, "왜 매칭되었는지" (공통으로 반응한 게시물이 무엇인지)를 보여주지 않음. 대화의 시작점(ice-breaker)이 없음
- **단방향 수락 구조**: 매칭이 생기면 user1, user2 모두 수락/거절 가능하지만, 한 쪽만 수락하면 바로 ACCEPTED가 됨. 쌍방 합의가 아닌 일방 수락 구조

## 2. 차별성 분석 (기존 서비스 대비)

| 비교 항목 | 틴더 | 블라인드 | 에브리타임 | **taste-match** |
|----------|------|---------|-----------|----------------|
| 매칭 기준 | 외모(사진) | 직장 인증 | 학교 인증 | **취향 공감도** |
| 익명성 | 낮음 | 중간(직장 공개) | 중간(학교 공개) | **높음(완전 익명)** |
| 진입 장벽 | 전화번호/SNS | 직장 이메일 | 학교 이메일 | **없음(원클릭)** |
| 콘텐츠 | 프로필 | 직장 이야기 | 학교 이야기 | **취향 게시물** |
| 대화 시작 | 매칭 즉시 | 댓글 | 댓글/쪽지 | **매칭 수락 후** |

**차별점 요약:**
1. **완전 익명 + 제로 진입장벽**: 회원가입 없이 쿠키 기반 세션으로 즉시 참여. 이것은 명확한 강점
2. **취향 기반 매칭**: 외모나 소속이 아닌 콘텐츠 공감도 기반. 기존 매칭 앱과 근본적으로 다른 접근
3. **리액션이 곧 매칭 시그널**: 별도의 "좋아요/스와이프" 없이 자연스러운 콘텐츠 소비 행위가 매칭으로 이어짐

**신선함 평가: B+**
- 컨셉은 신선하나, 실제 구현에서 그 신선함이 충분히 체감되지 않음
- "왜 이 사람과 매칭됐는지"를 보여주지 않으면 일반 랜덤 채팅과 차별화가 약해짐

## 3. UX 흐름 분석

### 사용자 여정 맵

```
[랜딩] → [익명 참여] → [피드 진입] → [게시물 작성/리액션] → [매칭 탭] → [수락/거절] → [1:1 채팅]
  index     /join         /feed        /posts/new, /react      /matches    /accept      /chat/{id}
```

**단계별 분석:**

#### (1) 랜딩 → 참여 (index.html → /join)
- **우수**: 원클릭 진입, 설명이 명확 ("가입 불필요, 개인정보 수집 없음")
- **문제**: 세션 만료(7일) 후 재진입 시 새 사용자로 취급됨. 기존 매칭/대화 기록 접근 불가

#### (2) 피드 (feed.html)
- **우수**: 카테고리 필터, 상대시간 표시, FAB(플로팅 버튼)로 게시물 작성 유도
- **문제**: 정렬이 최신순 고정 (`findAllByOrderByCreatedAtDesc`). 인기순/반응많은순 없음
- **문제**: 자기 게시물에도 리액션 가능 (자기 반응 차단 로직 없음)
- **문제**: 페이지네이션/무한스크롤 없음. 게시물 많아지면 성능 저하

#### (3) 게시물 작성 (post-form.html)
- **우수**: 실시간 미리보기, 글자수 카운터, 카테고리 선택 UI가 직관적
- **문제**: 작성 가이드/프롬프트 부족. "비오는 날 재즈 추천" 같은 placeholder만으로는 양질의 콘텐츠 유도가 어려움
- **문제**: 이미지 첨부 불가. 취향 공유에서 시각적 요소는 핵심인데 텍스트만 지원

#### (4) 리액션
- **우수**: 토글 방식 (같은 버튼 누르면 취소), 4종류 리액션 타입이 감정을 세분화
- **문제**: LIKE와 CURIOUS는 매칭에 반영되지 않음 (SAME_HERE, EMPATHY만 반영). 이 사실이 사용자에게 안내되지 않음
- **문제**: 리액션이 1인 1개로 제한됨 (`UniqueConstraint(post_id, reactor_id)`). "좋아요"와 "나도!" 동시 불가

#### (5) 매칭 (matches.html)
- **우수**: 상태별 UI 분리(PENDING/ACCEPTED/DECLINED), 매칭 점수 표시
- **치명적 문제**: 매칭 계산이 `/matches` 페이지 진입 시에만 실행됨. 매칭 탭을 안 열면 영원히 매칭이 생기지 않음
- **문제**: score >= 2로 매칭 기준이 너무 낮음. 2개의 게시물에 같은 리액션만으로 매칭
- **문제**: 매칭 이유(공통 취향 게시물)를 보여주지 않음

#### (6) 채팅 (chat.html)
- **우수**: 3초 폴링으로 실시간에 가까운 경험, Enter 전송, 상대 시간 표시
- **문제**: 읽음 표시 없음
- **문제**: unreadCount 계산이 부정확 — 상대방의 전체 메시지를 세고 있음 (내 마지막 메시지 이후가 아님)
- **문제**: 채팅방 나가기/차단 기능 없음

## 4. 핵심 문제점 상세

### 4.1 매칭 알고리즘의 한계

**현재 로직** (`MatchService.calculateMatches`):
- SAME_HERE 또는 EMPATHY 반응을 공유하는 게시물 수를 카운트
- score >= 2이면 매칭 생성

**문제점:**
1. **score 2는 너무 낮다**: 초기 사용자가 적을 때는 의미 있지만, 유저/게시물이 늘면 거의 모든 사람과 매칭됨
2. **카테고리 가중치 없음**: "음식"에서 2개 겹치는 것과 "음식+음악+영화"에서 3개 겹치는 것의 질적 차이를 반영하지 않음
3. **리액션 타입 가중치 없음**: SAME_HERE("나도!")가 EMPATHY("공감")보다 취향 일치도가 높지만 동일 점수
4. **시간 감쇠 없음**: 1달 전 반응과 오늘 반응이 동일 가중치
5. **매칭 포화 문제**: 기존 매칭 여부만 체크하고 최대 매칭 수 제한이 없음

### 4.2 카테고리 구성

현재 카테고리: `["음식", "음악", "영화", "패션", "장소", "취미", "기타"]`

**문제점:**
- "취미"가 너무 넓음 (게임, 운동, 독서 등이 모두 포함)
- "책/독서", "드라마/예능", "게임", "여행", "운동/스포츠" 등이 빠져 있음
- 카테고리가 하드코딩되어 있어 확장이 어려움 (`FeedController.CATEGORIES`)
- 복수 카테고리 선택 불가 (게시물 1개 = 카테고리 1개)

### 4.3 콘텐츠 유도 부족

- 온보딩 과정에서 취향 프로필 작성 단계가 없음
- 새 사용자가 빈 피드를 보고 무엇을 써야 할지 모를 수 있음
- 게시물 작성 시 카테고리별 예시/프롬프트가 하나뿐 (placeholder 1개)
- 트렌딩/인기 게시물 노출이 없어 참고할 콘텐츠가 부족

### 4.4 사용자 리텐션 메커니즘 부재

- 알림 시스템 없음 (매칭, 새 메시지, 내 게시물 반응)
- 일일 추천/큐레이션 없음
- 활동 보상(뱃지, 레벨 등) 없음
- 세션 만료 시 복구 불가

### 4.5 안전성/신뢰 문제

- 신고/차단 기능 없음
- 부적절 콘텐츠 필터링 없음
- 쿠키 기반 인증이라 쿠키 복사 시 세션 탈취 가능
- 세션 토큰이 HttpOnly가 아님 (`new Cookie("TASTE_TOKEN", ...)` — JS에서 접근 가능)

## 5. 개선 제안 (8개)

### 제안 1: 매칭 알고리즘 고도화
- **현재**: score >= 2 (공통 반응 게시물 2개 이상)
- **개선안**:
  - 카테고리 다양성 보너스: 서로 다른 카테고리에서 겹치면 가중치 부여 (예: 음식+음악 = 3점, 음식+음식 = 2점)
  - 리액션 타입별 가중치: SAME_HERE = 2점, EMPATHY = 1점
  - 동적 임계값: 전체 유저 수 대비 상위 N%만 매칭되도록 임계값 자동 조정
  - 매칭 상한선: 사용자당 동시 활성 매칭 최대 10개
- **난이도**: 중 | **임팩트**: 높음

### 제안 2: 매칭 컨텍스트 제공 (Ice-breaker)
- **현재**: 매칭 카드에 파트너 닉네임과 점수만 표시
- **개선안**:
  - 매칭 이유 표시: "이 사람과 공통으로 반응한 취향" 목록 노출
  - 공통 게시물 미리보기 (최대 3개)
  - 자동 생성 대화 시작 주제: "두 분 다 '비 오는 날의 재즈'에 공감했어요!"
- **난이도**: 중 | **임팩트**: 높음 (대화 전환율 직접 영향)

### 제안 3: 매칭 계산의 비동기화
- **현재**: `/matches` 페이지 진입 시 동기적으로 계산
- **개선안**:
  - 리액션 생성 시 이벤트 기반으로 매칭 후보를 갱신 (`@EventListener` 또는 `@Async`)
  - 또는 스케줄러로 주기적 계산 (`@Scheduled(fixedRate = 60000)`)
  - 새 매칭 발생 시 피드 상단 또는 네비게이션에 알림 뱃지 표시
- **난이도**: 중 | **임팩트**: 높음

### 제안 4: 온보딩 취향 프로파일링
- **현재**: 가입 후 바로 빈 피드로 이동
- **개선안**:
  - 가입 직후 "취향 퀴즈" 3~5문항 (예: "좋아하는 음식 장르는?", "주말에 주로 뭐해요?")
  - 응답 기반으로 관련 게시물 추천 피드 제공
  - 기존 인기 게시물 중 카테고리별 1개씩 "이런 취향 공유가 있어요" 안내
- **난이도**: 중 | **임팩트**: 높음 (초기 이탈 방지)

### 제안 5: 콘텐츠 유도 강화
- **현재**: 게시물 작성 시 placeholder 1개만 존재
- **개선안**:
  - 카테고리별 동적 프롬프트 제공 (음식: "인생 라면은?", 음악: "요즘 반복 재생하는 노래는?")
  - "오늘의 취향 질문" 데일리 프롬프트 (관리자 또는 자동 생성)
  - 카테고리 세분화: "취미" → "게임", "운동", "독서" 등으로 분리
  - 카테고리를 DB 관리로 전환하여 유연한 추가/수정 가능하게
- **난이도**: 낮~중 | **임팩트**: 중

### 제안 6: 알림 시스템 구축
- **현재**: 알림 없음. 사용자가 직접 확인해야 함
- **개선안**:
  - 네비게이션 바에 알림 뱃지 (새 매칭, 새 메시지, 내 게시물 반응)
  - 피드 상단 알림 배너 ("새로운 매칭 2건이 있어요!")
  - 장기적으로 웹 푸시 알림 (Service Worker)
- **난이도**: 중 | **임팩트**: 높음 (리텐션 핵심)

### 제안 7: 쌍방 매칭 수락 구조로 전환
- **현재**: 한 쪽이 수락하면 바로 ACCEPTED
- **개선안**:
  - 매칭 상태를 PENDING → USER1_ACCEPTED / USER2_ACCEPTED → MATCHED로 세분화
  - 양쪽 모두 수락해야 대화 가능
  - 수락 대기 중인 상대에게 "상대가 관심을 보이고 있어요" 알림
- **난이도**: 중 | **임팩트**: 중 (매칭 품질 향상, 원치 않는 대화 방지)

### 제안 8: 피드 정렬/필터 다양화
- **현재**: 최신순 고정, 카테고리 필터만 존재
- **개선안**:
  - 정렬 옵션: 최신순 / 반응 많은 순 / 랜덤
  - 내가 아직 반응하지 않은 게시물만 보기 필터
  - "취향이 비슷한 사람의 게시물" 개인화 피드 (매칭된 사람들의 게시물 우선 노출)
- **난이도**: 낮 | **임팩트**: 중

| 항목 | 점수 (5점 만점) | 코멘트 |
|------|:---:|--------|

| 콘텐츠 유도 | 2.5 | 온보딩과 프롬프트가 부족하여 초기 콘텐츠 빈곤 우려 |

| **종합** | **2.8** | **MVP로서 핵심 루프는 작동하나, 서비스 품질을 위해 매칭 고도화와 리텐션 장치가 시급** |

## 7. 우선순위 로드맵 (권장)

### Phase 1 — 즉시 (핵심 경험 개선)
1. 매칭 계산 비동기화 (제안 3)
